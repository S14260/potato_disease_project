# 面试问答准备 — 马铃薯病害多Agent智能诊断系统

## 一、项目整体

### Q1: 用一句话介绍你的项目？

基于 LangGraph Multi-Agent 架构的马铃薯病害智能诊断系统，用户上传叶片图片，系统自动完成病害检测、环境感知、风险评估、知识检索，生成治疗建议和动态防治工作流。

### Q2: 项目的技术栈是什么？

后端：Python + Flask + LangGraph + LangChain
AI：YOLOv8n 目标检测 + mimo-v2.5-pro LLM + ChromaDB RAG
前端：自定义 CSS + Chart.js + Lucide Icons
外部服务：OpenWeatherMap 气象 API

### Q3: 项目的架构是怎样的？

分两层。快速模式（默认）是线性管道：YOLO → 标准化 → 字典查知识 → 天气API → 风险引擎 → LLM生成报告。Agent 模式是 Supervisor 编排三个子 Agent（诊断、风险、治疗），LLM 通过 Function Calling 自主调用工具。默认用快速模式保证性能，Agent 模式用于演示 Multi-Agent 能力。

---

## 二、YOLO 目标检测

### Q4: YOLO 模型是怎么训练的？

用 Ultralytics YOLOv8n 作为预训练基座，在自己的马铃薯病害数据集上微调。数据集包含 early_blight 和 late_blight 两个类别，训练 100 epochs，输入尺寸 640×640。训练脚本是 `train.py`。

### Q5: YOLO 的推理速度怎么样？

CPU 上约 100ms（前处理 60ms + 推理 12ms + 后处理 27ms），GPU 上更快。对于 Web 应用来说完全够用，用户感知延迟主要在 LLM 调用。

### Q6: 如果检测到两种病害怎么办？

当前取置信度最高的作为主病害。如果需要扩展，可以对每种病害分别生成诊断报告，但这会增加系统复杂度。

---

## 三、LLM 调用

### Q7: LLM 在系统中扮演什么角色？

两个角色：
1. **诊断报告生成**：根据检测结果、环境数据、风险评分、药剂信息，生成 Markdown 格式的结构化诊断报告
2. **工作流动态决策**：根据农户反馈和历史步骤，生成下一步防治建议（返回 JSON）

### Q8: 为什么选 mimo-v2.5-pro 而不是 GPT-4？

mimo-v2.5-pro 是小米 MiMo 平台提供的 OpenAI 兼容 API，国内访问稳定、成本低。架构上用 ChatOpenAI 接口，换模型只改配置不改代码。

### Q9: LLM 的 prompt 是怎么设计的？

诊断报告的 prompt 包含：病害信息（名称、病原、症状）+ 环境信息（温度、湿度、天气）+ 风险评估（等级、评分）+ 推荐药剂（名称、成分、浓度），要求输出表格化的 Markdown 报告。

工作流的 prompt 包含：病害类型、风险等级、环境信息、历史步骤和反馈，要求输出 JSON 格式的下一步建议，并根据反馈类型（效果明显/无改善/恶化）调整策略。

### Q10: LLM 返回格式不稳定怎么办？

工作流场景下 LLM 要求返回 JSON，但可能带 markdown 代码块标记或多余文字。代码中做了三层处理：先 `json.loads` 直接解析，失败后去掉 ``` 标记再解析，最后用括号匹配提取第一个完整 JSON 对象。解析失败则返回兜底建议。

### Q11: max_tokens 为什么设 2000？

mimo-v2.5-pro 是思考模型，内部推理过程也消耗 tokens。如果 max_tokens 太小（如 512），推理过程用完 tokens 后实际输出为空。设 2000 保证有足够的 tokens 给最终输出。

## 四、RAG 向量检索

### Q12: RAG 在项目中是怎么用的？

构建 ChromaDB 向量数据库，将病害信息、农药数据、SOP 流程转为 LangChain Document 对象，用 `BAAI/bge-small-zh-v1.5` 模型生成向量存入 ChromaDB。查询时用自然语言做语义检索，支持按病害类型、文档类型过滤。

### Q13: RAG 和直接查字典有什么区别？

字典只能 `disease_db["early_blight"]` 精确匹配 key。ChromaDB 支持语义搜索，输入"叶子长斑用什么药"能匹配到早疫病的药剂信息。新增病害只需添加文档，不改代码。

### Q14: 当前 RAG 在哪个环节被调用？

在 Agent 模式下，Treatment Agent 调用 `search_knowledge_base` 工具时使用 RAG。快速模式下直接查 Python 字典，不走 RAG。

### Q15: 为什么用 bge-small-zh 而不是其他 Embedding 模型？

三个原因：(1) 中文优化，对农业术语有较好的语义理解；(2) 本地运行，不需要外部 API；(3) 模型体积小（~100MB），推理快。

### Q16: 数据量这么小，有必要用向量数据库吗？

功能上字典够用。但向量数据库的价值是：(1) 架构可扩展，新增病害只加文档；(2) 支持语义搜索；(3) 作为简历项目展示了完整 RAG 技术栈。

## 五、Multi-Agent 架构

### Q17: 为什么用 Multi-Agent 而不是单 Agent？

单 Agent 的 system prompt 要描述所有工具（7个）的使用场景，工具集过大会降低 LLM 选择准确率。分 Agent 后每个 prompt 聚焦一个领域，工具集 2-3 个，LLM 决策更准确。另外可以独立测试和优化每个 Agent 的 prompt。

### Q18: Supervisor 怎么决定调用顺序？

Supervisor 的 system prompt 描述了各 Agent 的能力和推荐顺序（诊断→风险→治疗）。LLM 根据 prompt 自主决策。prompt 中也指定了默认顺序作为保底。

### Q19: 为什么默认用快速模式而不是 Agent 模式？

Agent 模式下每个 Agent 内部可能多次调用 LLM（ReAct 循环：思考→调工具→再思考），加上 Supervisor 本身的 LLM 调用，一次诊断可能 5-8 次 LLM 调用，耗时 15 秒以上。快速模式只有 1 次 LLM 调用，3 秒内完成。用户体验差距很大。

### Q20: Agent 模式和快速模式的输出有什么区别？

功能上等价——都是病害检测 + 风险评估 + 治疗建议。区别在于 Agent 模式下 LLM 有更多自主权，可以决定调用哪些工具、调用几次，输出可能更个性化。快速模式是固定流程 + 单次 LLM 调用。

## 六、Function Calling / Tool Calling

### Q21: @tool 装饰器做了什么？

读取函数的 docstring 作为工具描述（LLM 看到的 description），读取 type hints 生成 JSON Schema（定义参数类型和必填项），包装成 LangChain 的 BaseTool 对象。最终 LLM 看到的是标准的 OpenAI Function Calling 格式。

### Q22: 一共有哪些工具？

| 工具名 | 功能 | 底层实现 |
|--------|------|---------|
| detect_disease | YOLO 病害检测 | yolo/predict.py |
| get_disease_info | 病害信息查询 | knowledge/disease_db.py |
| get_weather | 气象数据获取 | utils/weather_api.py |
| compute_risk_score | 风险评分计算 | engine/decision_engine.py |
| search_knowledge_base | RAG 语义检索 | rag/retriever.py |
| get_sop | SOP 流程查询 | knowledge/sop_db.py |
| generate_treatment_advice | LLM 治疗建议 | ai/llm_agent.py |

### Q23: 如果 LLM 调用了错误的工具怎么办？

三层防御：(1) 每个 Agent 的 system prompt 限定工具使用场景和前置条件；(2) 工具函数内部参数校验，返回结构化错误；(3) Supervisor 有 max_rounds 限制防止循环调用。

## 七、风险评估引擎

### Q24: 风险评分是怎么计算的？

规则引擎，多因子加分：
- 病斑数量：≥20 → +40分，≥10 → +25分，≥5 → +15分
- 平均置信度：≥80% → +20分，≥60% → +10分
- 湿度：≥85% → +25分，≥70% → +15分
- 降雨：有 → +20分
- 温度适宜度：早疫病 20-30℃ → +15分，晚疫病 15-22℃ → +20分
- 生长期：块茎期 → +15分，生长期 → +10分

总分 <40 低风险，40-69 中风险，≥70 高风险。

### Q25: 为什么用规则引擎而不是 ML 模型？

(1) 数据量太小，训练不了 ML 模型；(2) 规则引擎可解释，每个评分因子都有明确原因，方便向农户解释；(3) 规则可以根据农业专家知识调整，不需要标注数据。

## 八、防治工作流

### Q26: 工作流是怎么实现的？

SOP 标准流程 + LLM 动态调整。`knowledge/sop_db.py` 定义了每个病害+风险等级的标准步骤序列。用户完成一步后提交反馈，LLM 根据反馈类型生成下一步建议。反馈「效果明显」→ 进阶步骤，「无改善」→ 加强措施，「恶化」→ 紧急方案。

### Q27: 工作流和普通的 if-else 有什么区别？

if-else 是固定的决策树，只能处理预设的情况。LLM 动态生成的优势是：(1) 可以结合环境数据（温度、湿度）生成针对性建议；(2) 可以综合历史反馈做出判断；(3) 措辞自然，农户更容易理解。

### Q28: LLM 生成的工作流步骤可靠吗？

有保障机制：(1) SOP 提供初始框架，LLM 在此基础上调整，不是完全自由发挥；(2) prompt 中明确了反馈→策略的映射关系；(3) 兜底机制：LLM 调用失败时返回默认建议。

## 九、前端

### Q29: 前端用了什么技术？

自定义 CSS 框架（CSS 变量 + Grid + Flexbox），Chart.js 做数据可视化，Lucide Icons 做图标，Jinja2 模板引擎渲染页面。没有用 React/Vue，因为是 Flask 项目，服务端渲染足够。

### Q30: 前端有哪些页面？

7 个页面：Dashboard 数据看板、病害诊断、防治工作流、历史记录、病害知识库、数据分析、系统设置。通过左侧菜单栏切换，响应式布局适配移动端。

## 十、工程实践

### Q31: 错误处理怎么做的？

三个层面：
1. **工具层面**：每个工具函数 try-catch，返回错误信息而非抛异常
2. **Agent 层面**：max_rounds 限制防止无限循环
3. **系统层面**：LLM 调用失败时回退到兜底方案（`_fallback_advice()`），确保用户总能得到结果

### Q32: 怎么扩展到更多病害？

三步：(1) `knowledge/*.py` 添加新病害数据；(2) 运行 `python rag/ingest.py` 更新向量数据库；(3) 如需新 YOLO 类别，重新训练模型。Agent 层面零修改。

### Q33: 有什么可以改进的地方？

1. 加入数据库（SQLite/PostgreSQL）替代 JSON 文件存储历史
2. 添加用户认证系统
3. YOLO 模型换成更大的 YOLOv8m/l 提升精度
4. RAG 检索加入 Reranker 二次排序
5. 工作流加入人工确认节点（Human-in-the-loop）
6. 部署方案：Docker + Nginx + Gunicorn

### Q34: 这个项目你学到了什么？

1. Agent 系统不是工具越多越好，拆分 Agent 降低单个 Agent 的复杂度更有效
2. RAG 在数据量小时价值不在检索精度，而在架构可扩展性
3. LLM 输出不稳定是工程问题，需要 robust 的解析和兜底机制
4. 用户体验和 AI 能力需要平衡，快速模式比 Agent 模式体验好很多

## 十一、多轮对话与状态管理

### Q35: 多轮对话怎么实现的？

三个机制配合：
1. **State 层**：DiagnosisState 中定义 `conversation_history` 字段，存储历史诊断摘要（病害、风险等级、城市、温湿度）
2. **持久化层**：LangGraph 的 MemorySaver Checkpointer + thread_id，每次诊断完成后 State 自动保存，同一 thread_id 的后续调用自动加载
3. **LLM 层**：`treatment_node` 读取 history，格式化后注入 prompt，LLM 可以引用"上次诊断为早疫病高风险"

### Q36: thread_id 怎么管理的？

Flask session 自动管理。用户首次访问时生成 `session-{uuid}` 写入 cookie，后续请求自动携带。同一浏览器的多次诊断共享同一个 thread_id，State 自动关联。也可以通过 API 参数手动指定 thread_id。

### Q37: Checkpointer 有什么用？

Checkpointer 是 LangGraph 用于持久化 Graph 执行状态的机制。它会保存当前 State、消息历史以及执行进度，使 Agent 在多轮对话、长任务执行或异常中断后能够恢复，而不是重新从起点执行。

在我的马铃薯病害诊断系统中，我通过 thread_id 为每个用户维护独立的诊断状态，这样用户可以在后续对话中继续当前诊断流程，而无需重新上传图片或重新执行已完成的步骤。

核心用途：
1. **多轮对话**：同一 thread_id 的 State 跨请求保留，LLM 可引用历史诊断上下文
2. **Human-in-the-loop**：在某个节点 `interrupt()` 暂停，等人工确认后 `resume()` 继续
3. **故障恢复**：进程崩溃后可从最近的 checkpoint 继续，不丢失已完成的步骤
4. **时间旅行**：回溯到任意 checkpoint 查看历史状态，方便调试

开发环境用 MemorySaver（内存），生产环境换 PostgresSaver（PostgreSQL），只需改一行代码，业务逻辑零修改。

### Q38: 为什么用 State 存 history 而不是直接用数据库？

因为 LangGraph 的 Checkpointer 已经自动持久化整个 State。把 history 放在 State 里，不需要额外的数据库，也不需要手动读写。而且 State 的更新是原子性的，避免了并发写入的数据一致性问题。

### Q39: 如果用户换了浏览器，历史还在吗？

不在。Flask session 基于 cookie，换浏览器就生成新的 thread_id，State 互相隔离。要实现跨设备的历史共享，需要加用户认证系统 + 数据库持久化 thread_id 的映射关系。当前版本是单机 demo，没有做这个。
