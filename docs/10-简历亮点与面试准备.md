# 简历亮点与面试准备

## 简历项目描述

### 项目名称
马铃薯病害多Agent智能诊断系统

### 时间
2026.03 - 2026.05

### 技术栈
Python、Flask、LangGraph、ChromaDB RAG、OpenAI Function Calling、ReAct Agent

### 项目描述
面向农业病害防治场景，构建集病害识别、环境风险评估、智能问诊与动态防治决策于一体的农业智能诊断平台，实现"病害检测—风险分析—知识检索—方案生成—动态干预"的业务闭环。

### 核心工作

1. 自行采集并标注马铃薯病害叶片图片，基于 YOLOv8 完成早疫病、晚疫病目标检测模型的训练与部署
2. 基于 LangGraph Supervisor 模式设计多 Agent 协作架构，实现诊断、风险评估、治疗三个 Agent 的自主任务分派与执行
3. 构建 ChromaDB 向量知识库，实现病害、农药、SOP 数据的语义检索；基于 @tool 装饰器封装 7 个工具，LLM 自主选择调用
4. 开发基于 SOP 的动态防治工作流，结合环境数据与农户反馈自动生成下一阶段防治方案，实现病害管理流程闭环

### 项目亮点

1. **Multi-Agent 架构设计：** 采用 LangGraph Supervisor 模式，将诊断、风险评估、治疗三个领域解耦为独立 Agent，每个 Agent 拥有专属 system prompt 和工具集（2-3 个），通过编排 Agent 统一调度，避免单 Agent 工具集过大导致调用准确率下降
2. **RAG 知识增强：** 构建 ChromaDB 向量数据库，将病害、农药、SOP 数据向量化入库，支持自然语言语义检索（如"叶子长斑用什么药"匹配到早疫病信息），新增病害只需添加文档无需改代码
3. **Tool Calling 机制：** 基于 OpenAI 兼容协议实现 Function Calling，LLM 自主决定调用时机和参数；工具函数内置参数校验与结构化错误返回，Supervisor 设有 max_rounds 限制防循环调用，失败时自动回退到直接函数调用
4. **双模式架构：** 提供快速模式（直接函数调用 + LLM 报告生成，<5s 响应）和 Agent 模式（多 Agent 协作推理，完整 Tool Calling 链路），前端支持一键切换并可视化 Agent 协作流程

## 面试问答准备

### 架构相关

**Q: 为什么选择 Multi-Agent 而不是单个 Agent 加所有工具？**
A: 单 Agent 的 system prompt 会很长（需要描述所有工具和使用场景），工具集过大会导致 LLM 选择准确率下降。分 Agent 后每个 prompt 聚焦一个领域，工具集更小（2-3个），LLM 更容易做出正确决策。另外，分 Agent 后可以独立测试和优化每个领域的 prompt。

**Q: Supervisor 怎么决定调用顺序？**
A: Supervisor 的 system prompt 描述了各 Agent 的能力和推荐顺序（诊断→风险→治疗）。LLM 根据 prompt 和用户请求自主决策。同时，prompt 中明确指定了默认顺序作为保底，确保流程可靠性。

**Q: 为什么不用 LangChain 的 AgentExecutor？**
A: AgentExecutor 是旧版模式，LangGraph 的 create_react_agent 是 2025+ 的推荐方案。LangGraph 返回的是状态图（StateGraph），支持流式输出、断点续传、人工介入等高级特性，且可以直接组合成 Supervisor 多 Agent 系统。

### RAG 相关

**Q: 为什么用 ChromaDB 而不是直接用字典？**
A: 字典只能精确匹配 key，不支持模糊查询。ChromaDB 是向量数据库，支持语义搜索——用户问"叶子长斑用什么药"能匹配到"早疫病"的相关信息。另外，新增病害只需添加文档，不修改代码，扩展性更好。

**Q: Embedding 模型选的什么？为什么？**
A: 使用 HuggingFace 的 `BAAI/bge-small-zh-v1.5`，中文优化的 sentence-transformer 模型。选择理由：(1) 中文支持好，对农业术语有较好的语义理解；(2) 本地运行，不需要外部 API；(3) 模型体积小（~100MB），推理速度快。

**Q: 两个病害的数据量很小，有必要用向量数据库吗？**
A: 从功能上看，当前数据量确实可以用字典。但用向量数据库的价值在于：(1) 架构可扩展，新增病害只需添加文档；(2) 支持语义搜索，用户体验更好；(3) 作为简历项目，展示了 RAG 技术栈的完整实现。

### Tool Calling 相关

**Q: Function Calling 和普通的 API 调用有什么区别？**
A: 普通 API 调用是程序员写死调用逻辑（if-else 决定调什么函数）。Function Calling 是 LLM 根据用户意图自主决定调用哪个函数、传什么参数。程序员只需定义工具列表和描述，LLM 负责决策。这是 Agent 系统的核心能力。

**Q: 如果 LLM 调用了错误的工具怎么办？**
A: 三层防御：(1) 每个 Agent 的 system prompt 明确限定了工具使用场景和前置条件；(2) 工具函数内部有参数校验，返回结构化错误信息；(3) Supervisor 有 max_rounds 限制防止循环调用，超时后回退到直接函数调用。

**Q: @tool 装饰器做了什么？**
A: 它读取函数的 docstring 作为工具描述（LLM 看到的 description），读取 type hints 生成 JSON Schema（定义参数类型和必填项），然后包装成 LangChain 的 BaseTool 对象。最终 LLM 看到的是标准的 OpenAI Function Calling 格式。

### 工程相关

**Q: 为什么选择 mimo-v2.5-pro 而不是 GPT-4？**
A: mimo-v2.5-pro 是通过小米 MiMo 平台提供的 OpenAI 兼容 API，支持 Function Calling 协议，且在国内访问稳定、成本更低。架构上使用 ChatOpenAI 接口，换模型只需改配置，不改代码。

**Q: 错误处理怎么做的？**
A: 三个层面：(1) 工具层面：每个工具函数内部 try-catch，返回错误信息而非抛异常；(2) Agent 层面：max_rounds 限制防止无限循环；(3) 系统层面：Supervisor 失败时回退到直接调用原始函数，确保用户总能得到结果。

**Q: 这个系统怎么扩展到更多病害？**
A: 三步：(1) 在 knowledge/*.py 中添加新病害的数据；(2) 重新运行 `python rag/ingest.py` 更新向量数据库；(3) 如果需要新的 YOLO 类别，重新训练模型。Agent 层面不需要任何修改，因为工具是通用的。
